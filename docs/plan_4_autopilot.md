# Plan 4 — Autopilot Feature

## Overview

OpenHelm's Autopilot feature adds a strategic autonomy layer on top of the existing tactical self-correction system. Users can configure three levels of automation via a single master switch, controlling both per-run self-correction and proactive system-generated monitoring jobs.

---

## Architecture

### Two Layers of Autonomy

```
Autopilot Mode (master switch)
├── OFF → no self-correction, no system jobs, fully manual
├── APPROVAL REQUIRED → self-correction ON, system job proposals need inbox approval
└── FULL AUTO → self-correction ON, system jobs created & run automatically
```

**Tactical layer** (existing): Per-run self-correction in `executor/self-correction.ts`. When a job fails, Haiku analyzes logs and optionally retries with correction guidance (max N retries). Now gated by `autopilot_mode` instead of the deprecated `auto_correction_enabled` setting.

**Strategic layer** (new): System-generated Claude Code jobs that proactively monitor goal health and assess progress. Generated by Sonnet, run on Haiku for cost efficiency.

---

## Data Model Changes

### New columns on `jobs` table
- `source TEXT NOT NULL DEFAULT 'user'` — `'user'` | `'system'`
- `system_category TEXT` — free-form lowercase_snake_case string (e.g. `health_watchdog`, `progress_review`). Only set when `source = 'system'`. Used for deduplication and display grouping.

### New setting: `autopilot_mode`
- Values: `"full_auto"` | `"approval_required"` | `"off"`
- Default (no stored value): `full_auto`
- Replaces `auto_correction_enabled` (deprecated)

### New table: `autopilot_proposals`
Stores pending system job proposals in `approval_required` mode before user acts on them.

```sql
CREATE TABLE autopilot_proposals (
  id TEXT PRIMARY KEY,
  goal_id TEXT NOT NULL REFERENCES goals(id) ON DELETE CASCADE,
  project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | approved | rejected | expired
  planned_jobs TEXT NOT NULL,              -- JSON: PlannedSystemJob[]
  reason TEXT NOT NULL,
  created_at TEXT NOT NULL,
  resolved_at TEXT
);
```

## System Job Generation

### Flow
1. After a goal is created (via chat `handleApproveAll` or explicit `autopilot.generateForGoal` IPC call)
2. `generateAndHandleSystemJobs(goalId, projectId)` in `agent/src/autopilot/index.ts` is invoked
3. If mode is `off`, returns early
4. `generateSystemJobs()` in `agent/src/planner/system-jobs.ts` calls Sonnet via `callLlmViaCli()`
5. LLM returns 1–3 `PlannedSystemJob` objects with `systemCategory`, schedule, and prompt
6. If `full_auto`: jobs created immediately (Haiku, low effort)
7. If `approval_required`: `AutopilotProposal` record created, `autopilot.proposalCreated` event emitted

### Cost Controls
- System jobs default to `model: "claude-haiku-4-5-20251001"`, `modelEffort: "low"` (~10-20x cheaper than Sonnet)
- Planner prompt enforces weekly minimum schedules (never more than daily)
- Maximum 3 system jobs per goal
- System job prompts request concise 1–3 paragraph output

### System Job Prompt Guidelines
- Jobs OBSERVE and REPORT — they do not modify code unless explicitly appropriate
- Focus areas: health monitoring (failed runs, token spikes, permission issues) and progress assessment
- Each prompt must be fully self-contained

## Guardrails

### Token Budget (20% cap)
In the scheduler tick (`agent/src/scheduler/index.ts`), before enqueuing a system job:
1. Sum total system job token usage for the goal
2. Sum total user job token usage for the goal
3. If user usage > 1,000 tokens AND system usage > 20% of user usage:
   - Disable the system job (`isEnabled = false`)
   - Create an `autopilot_limit` inbox item notifying the user

This prevents system jobs from consuming a disproportionate share of the goal's token budget.

### Recursion Depth Guard
System jobs cannot trigger generation of more system jobs. Enforced by:
- Only calling `generateAndHandleSystemJobs` from user-facing goal creation flows
- The system-jobs planner only receives user-created jobs as context (system jobs excluded)
- Self-correction (tactical retries) IS still allowed on system jobs — they're real jobs

### Escalation
When limits are hit, an `autopilot_limit` inbox item is created with usage statistics.

## Self-Correction Integration

`agent/src/executor/self-correction.ts` previously checked `getSetting("auto_correction_enabled")`. Now checks `autopilot_mode`:

```typescript
const autopilotMode = getSetting("autopilot_mode");
if (autopilotMode?.value === "off") {
  return { attempted: false, reason: "Autopilot is off — self-correction disabled" };
}
```

When the user changes `autopilot_mode` to `"off"` via settings:
- All system jobs are disabled (`isEnabled = false`)
- All pending proposals are expired
- `autopilot.modeChanged` event is emitted to the frontend

## Frontend UI Changes

### Settings Screen
New `AutopilotSection` component (`src/components/settings/autopilot-section.tsx`) added between Execution and Application sections. Contains:
- Select control for Off / With Approvals / Full Auto with contextual description text
- Max correction retries setting (shown only when mode ≠ off)

The old `auto_correction_enabled` toggle and its max-retries dropdown were removed from `ExecutionSection`.

### Onboarding Wizard
New step 6 (`src/components/onboarding/steps/autopilot-step.tsx`) added before payment/complete. Presents a RadioGroup with three options. Full Auto is the recommended default. The wizard grew from 8 to 9 steps.

### Sidebar
`src/components/layout/sidebar-job-node.tsx`: System jobs display a `Bot` icon from lucide-react after the job name, providing subtle visual distinction.

### Goal Detail View
`src/components/content/goal-detail-view.tsx`:
- User jobs displayed under "Jobs (N)" subsection
- System jobs displayed under "System Jobs (N)" subsection with an "Autopilot" badge
- Both sections use a shared `JobsTable` helper component

### Inbox View
`src/components/content/inbox-view.tsx`:
- Proposals section displayed above alerts when pending proposals exist
- Shows proposal cards with job list, Approve (check icon), and Dismiss (X icon) buttons
- On approve: calls `approveAutopilotProposal`, reloads proposals
- On dismiss: calls `rejectAutopilotProposal`, reloads proposals
